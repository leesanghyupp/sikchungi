> 최종 수정: 2026-05-11

## 1. 프로젝트 개요

### 1.1 한 줄 소개
**"뭐 먹지?"** 라는 일상의 고민에, 사용자의 상황을 듣고 음식 3가지를 이유와 함께 추천해주는 AI 에이전트

### 1.2 프로젝트명
**`식충이`** ✅
- 컨셉: 먹는 거 좋아하는 친근한 캐릭터
- 도메인 후보: `sikchungi.app` / `sikchungi.vercel.app`

### 1.3 배경 / 동기
- "뭐 먹지"는 거의 매일 마주치는 사소하지만 피곤한 결정
- 단순 랜덤 추천이 아니라, 지금 기분·시간·상황에 맞는 **맥락 기반 추천**이 필요함
- AI 에이전트의 핵심 패턴(컨텍스트 수집 → 추론 → 구조화 출력)을 학습하기에 적합한 규모

### 1.4 학습 목표 (토이 프로젝트 관점)
- **Gemini API + structured output** 실전 적용 (한국 결제 이슈로 Claude 대신 Gemini 채택)
- Context caching으로 비용/속도 최적화 경험
- 대화형 에이전트 UX 설계
- Next.js 15 App Router + Vercel 배포 흐름 복습
- (선택) 추후 Claude로 교체 가능하도록 LLM 호출 로직 분리

## 2. 목표

### 2.1 사용자 입장 핵심 가치
- **결정 피로 최소화** — 30초 안에 만족스러운 답을 받음
- **납득 가능성** — "왜 이걸 추천했는지" 이유가 보임
- **재시도 친화적** — 마음에 안 들면 즉시 다른 추천

### 2.2 성공 기준 (MVP)
- [ ] 사용자가 3~5턴 안에 추천을 받을 수 있음
- [ ] 추천된 3개 음식 각각에 추천 이유가 붙음
- [ ] "이거 말고 다른 거" 재추천 동작
- [ ] 모바일 + 데스크톱에서 자연스럽게 동작
- [ ] Vercel에 배포되어 URL로 접근 가능

## 4. 핵심 기능 (MVP)

### 4.1 대화형 컨텍스트 수집 ⭐
에이전트가 폼이 아닌 **대화로** 사용자 상황을 캐물음

수집할 핵심 정보:
| 항목 | 예시 |
|---|---|
| 시간대 | 아침 / 점심 / 저녁 / 야식 |
| 기분·컨디션 | 피곤함, 스트레스, 가벼움, 든든함 원함 |
| 선호 카테고리 | 한식 / 양식 / 일식 / 중식 / 상관없음 |
| 최근에 먹은 것 | 중복 회피 |
| 동행 여부 | 혼자 / 같이 (분위기 영향) |
| 제약 | 매운 거 X, 비건, 알레르기 등 (선택) |

> 💡 **핵심 트위스트**: 한 번에 다 묻지 않고, 에이전트가 1~2개씩 자연스럽게 캐물음. 사용자가 이미 제공한 정보는 다시 안 묻기.

### 4.2 음식 추천 (3가지)
출력 포맷:
```
1. 김치찌개
   추천 이유: 비 오는 저녁 + 든든한 거 원함 + 최근에 양식 많이 드셨다고 했으니
   대표 특징: 따뜻함, 매콤함, 밥과 함께

2. ...
3. ...
```

### 4.3 재추천
- "이거 말고 다른 거" 버튼 또는 자연어 입력
- 이전 추천 음식은 제외
- 컨텍스트 유지 (다시 묻지 않음)

### 4.4 새 대화 시작
- 새로 시작 버튼 → 컨텍스트 초기화

## 5. 사용자 플로우

```
[진입]
  ↓
에이전트 인사 + 첫 질문 ("지금 시간대가 어떻게 되세요?")
  ↓
사용자 답변
  ↓
에이전트가 다음 필요한 정보 1~2개 추가 질문
  ↓ (충분한 컨텍스트 수집 완료)
에이전트 → 음식 3개 추천 + 이유
  ↓
사용자 선택지:
  - 만족 → 종료 (또는 "다음에 또 봐요" 메시지)
  - 재추천 → 다른 3개
  - 추가 정보 ("매운 거 빼고") → 필터 반영해서 재추천
```

## 8. AI 에이전트 설계

### 8.1 에이전트 페르소나
- **이름**: 식충이
- **톤**: **잼민이 톤** ✅
  - 초딩스러운 말투, 어휘는 단순하고 직진
  - 반말 기본, 이모지 자주 사용 (🍔🍕🍜😋🤤🔥)
  - 과장된 리액션 ("헐 그럼 이거 무조건 ㄱㄱ", "이거 안 먹으면 손해임")
  - 줄임말/유행어 OK ("ㅇㅇ", "ㄹㅇ", "개꿀", "존맛", "ㄱㄱ")
  - 단, 너무 알아듣기 힘들 정도는 X — 추천 자체는 명확해야 함
- **태도**: 사용자가 굶주린 동생/친구라고 생각하고 흥분해서 추천해주는 느낌
- **샘플 대사**:
  - "ㅇㅋ 점심임? 지금 배 얼마나 고픔? 살짝? 아니면 개고픔?? 🤤"
  - "헐 그럼 이거 ㄹㅇ 추천 ㄱㄱ — 김치찌개 먹어 진짜로"
  - "양식 아까 먹었다며 ㅋㅋ 그럼 이건 패스고 다른 거 줄게"

### 8.2 시스템 프롬프트 골격 (의사 코드)
```
너는 "식충이"라는 음식 추천 에이전트야.

[캐릭터]
- 먹는 거 진짜 좋아하는 잼민이(초딩) 톤
- 반말, 이모지 적극 사용 (🍔🍜😋🤤🔥)
- 흥분한 듯한 리액션 ("헐", "ㄹㅇ", "개꿀", "ㄱㄱ")
- 단, 추천 자체와 이유는 명확하게

[목표]
사용자가 지금 뭘 먹어야 할지 고민할 때, 상황을 빠르게 파악하고
음식 3가지를 이유와 함께 추천한다.

[행동 원칙]
1. 한 번에 모든 걸 묻지 마. 1~2개씩 자연스럽게 물어봐.
2. 충분한 정보 모았으면 바로 추천 (4~5턴 이내).
3. 추천 이유에는 "이 사람한테 왜 이게 맞는지" 맥락이 들어가야 함.
4. 사용자가 이미 답한 정보는 다시 묻지 마.
5. 잼민이 톤 유지하되, 너무 알아듣기 힘들 정도는 X.

[출력 형식]
- 일반 대화: 자연어 (잼민이 톤)
- 추천 시: structured output (JSON)으로 음식 3개 + 이유 + 태그 + 이미지 검색어
```

### 8.3 출력 구조 (Tool use 또는 structured output)
```typescript
type Recommendation = {
  status: "asking" | "recommending";

  // status === "asking"
  message?: string;

  // status === "recommending"
  recommendations?: {
    name: string;            // "김치찌개"
    reason: string;          // 잼민이 톤 추천 이유
    tags: string[];          // ["따뜻함", "매콤함"]
    image_query: string;     // Unsplash 검색어 (영문 권장: "kimchi stew")
  }[];
  context_summary?: string;  // "점심 / 든든함 원함 / 한식 선호"
};
```

> 💡 `image_query` 필드를 Claude가 직접 생성 → Unsplash API 검색에 사용

### 8.4 비용 최적화 전략
- **Gemini 2.5 Flash** 기본 사용 (무료 티어: 분당 15req, 일일 1,500req)
- 무거운 추론 필요 시에만 Gemini 2.5 Pro 호출
- 시스템 프롬프트 길이 최소화 + structured output으로 출력 토큰 제한
- (확장) Gemini Context Caching API로 멀티턴 대화 비용 절감

- **컨셉**: 동그랗고 통통한 주황색 캐릭터, 항상 입맛 다시고 있음
- **특징**:
  - 큰 눈 (반짝이 포함) → 호기심·기대감
  - 벌어진 입 + 분홍 혀 → "먹고 싶어" 표현
  - 양쪽 볼 홍조 → 친근함, 잼민이 느낌
  - 머리 위 김(🌫️) → 음식 냄새 맡고 흥분한 느낌
  - 작은 팔 → 귀여움
- **활용 방안**:
  - 헤더/로고
  - 에이전트 메시지 옆 아바타
  - 로딩 인디케이터 (점프하는 식충이 등)
  - OG 이미지
